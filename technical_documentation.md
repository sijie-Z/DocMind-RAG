## 1. 系统概述

本项目是一个设计精良、功能完备的企业级AI知识库系统。它基于FastAPI框架构建，并集成了一整套现代化的技术栈，包括对象存储、多级缓存、消息队列、混合搜索引擎和关系型数据库，形成了一个高内聚、可扩展的后端服务。

其核心能力是提供基于**检索增强生成 (RAG)** 的智能问答，并实现了诸多业界领先的优化策略。

## 2. 核心技术栈 (基于`.env`的真实配置)

- **Web框架**: FastAPI
- **数据库**: MySQL/PostgreSQL (通过SQLAlchemy ORM，支持异步aiomysql)
- **搜索引擎**: 
    - **关键词搜索**: Elasticsearch
    - **向量搜索**: FAISS
- **缓存**: Redis (用于数据库查询缓存) + In-Memory Cache (用于RAG结果缓存)
- **消息队列**: Kafka (用于服务间异步通信，如文档处理)
- **对象存储**: MinIO (用于存储上传的原始文件)
- **AI模型 (智谱AI全家桶)**: 
    - **对话LLM**: **`glm-4-flash`**
    - **向量Embedding**: **`embedding-3`** (向量维度: **2048**)
    - **重排Reranker**: **`rerank`** (已启用)
    - **API端点**: `https://open.bigmodel.cn/api/paas/v4/`

## 3. RAG核心流程详解

系统实现了一套非常先进的RAG流程，可以概括为 **“查询理解 -> 混合检索 -> 重排 -> 压缩 -> 生成”** 五个阶段。

### 3.1. 数据处理管道 (Ingestion)

1.  **文件上传**: 用户通过API上传文档，文件被存储到MinIO。
2.  **任务分发**: 后端服务向Kafka发送一个“新文档待处理”的消息。
3.  **异步处理 (Worker)**: 后台的Worker服务消费该消息，执行以下操作：
    a. **文档解析**: 从MinIO下载文件，使用`doc_parser`解析文本内容。
    b. **文本分块**: 将长文本切割成小的`DocumentChunk`。
    c. **向量化**: 调用`EmbeddingService`，使用**智谱`embedding-3`模型**将每个文本块转换为**2048维**的向量。
    d. **建立索引**: 
        - 将文本块内容和元数据存入**Elasticsearch**，以支持关键词搜索。
        - 将向量存入**FAISS**，以支持相似度搜索。

### 3.2. 问答响应管道 (Inference)

当用户发起一个查询时，系统执行以下步骤 (`rag_service.py`):

1.  **多级缓存检查**: 
    - **精确缓存**: 检查内存中是否存在完全相同的查询结果，如果命中则直接返回。
    - **语义缓存**: 如果精确缓存未命中，则计算查询向量，检查是否存在语义相似的查询结果，如果命中则直接返回。

2.  **查询理解 (`QueryIntentClassifier`)**: 
    - **目的**: 识别用户的查询意图（如事实型、流程型、定义型等）。
    - **实现**: 通过关键词匹配，为查询打上意图标签。

3.  **混合检索 (`HybridRetriever`)**: 
    - **关键词检索**: 根据上一步的意图分析，在Elasticsearch中进行**动态加权**的关键词搜索。
    - **向量检索**: 同时，将用户问题使用**智谱`embedding-3`模型**向量化，在FAISS中进行向量相似度搜索。
    - **结果融合 (RRF)**: 将两路检索结果使用`alpha`参数进行加权融合，计算出一个综合分数，生成一个初步的候选集。

4.  **重排 (`Reranker`)**: 
    - **目的**: 对初步的候选集进行二次排序，以提高最相关文档的排名。
    - **实现**: 调用**智谱`rerank`模型**，对候选集进行打分和重排序，筛选出最相关的Top-K个结果。

5.  **上下文压缩 (`ContextCompressor`)**: 
    - **目的**: 减少提供给大模型的上下文长度，去除噪音，提高信噪比。
    - **实现**: 提取与问题最相关的句子，将每个文档块压缩到预设的长度。

6.  **答案生成**: 
    - **Prompt构建**: 将压缩后的上下文、原始问题、以及可能的系统提示词（从`/prompts`模块获取）组装成最终的Prompt。
    - **LLM调用**: 将Prompt发送给**智谱`glm-4-flash`模型**，并以**流式响应 (SSE)** 的方式将答案实时返回给前端。

## 4. AI服务深度剖析 (Deep Dive)

### 4.1. 向量嵌入服务 (`embedding_service.py`)

这部分代码清晰地回答了“向量嵌入用的是什么？”的问题。

#### **4.1.1. 核心功能：智谱`embedding-3`**

-   根据您的`.env`配置，`EmbeddingService`会使用`EMBEDDING_API_KEY`连接到智谱的API端点(`https://open.bigmodel.cn/api/paas/v4/`)，并调用**`embedding-3`**模型进行向量化。代码中的`ENABLE_LOCAL_EMBEDDING`开关在您的环境中为`False`，因此不会使用Ollama。

#### **4.1.2. 工程鲁棒性：强大的容错机制**

-   **自动重试与指数退避**: `_embed_batch_with_retry`方法在调用智谱API时，如果遇到`429`(速率限制)、`403`(配额问题)或超时等临时性错误，会自动进行最多5次重试，并采用指数退避策略，极大地增强了服务的稳定性。

#### **4.1.3. 性能与成本：批处理与限流**

-   **批处理 (`batch_size`)**: 为了提升效率，服务会将待处理的文本列表切分为大小为8的小批次，批量调用智谱API。
-   **主动延迟**: 在处理多个批次之间，会主动`sleep(0.25)`秒，以避免触发服务端的速率限制。

### 4.2. RAG核心服务 (`rag_service.py`)

这是整个项目的“大脑”，包含了诸多高级的RAG优化技巧。

#### **4.2.1. 查询理解：`QueryIntentClassifier`**

-   **功能**: 在执行任何检索之前，系统首先会尝试理解用户的“意图”。它通过关键词匹配，将用户的问题分类为事实型、流程型、定义型、比较型等8种类型。
-   **作用**: 这是非常高级的优化。不同类型的问题，其答案可能分布在文档的不同位置。通过识别意图，系统可以在后续的Elasticsearch关键词检索中，**动态地为不同的字段设置不同的权重** (`get_intent_boost_fields`方法)。

#### **4.2.2. 上下文精炼：`ContextCompressor`**

-   **功能**: RAG的一个常见痛点是，检索到的上下文（Context）可能很长，包含大量与问题无关的“噪音”。这个压缩器就是为了解决此问题。
-   **实现**: 它首先将检索到的长文本切分成句子，然后计算每个句子与用户问题的相关性，优先保留高相关性的句子，从而精炼上下文。

#### **4.2.3. 重排器 (`Reranker`) 的具体实现**

-   **配置**: 您的`.env`中`RAG_ENABLE_RERANKER`为`true`，因此重排功能**已激活**。
-   **调用逻辑**: 在`_cross_encoder_rerank`方法中，由于您的`RERANK_API_URL`是`https://open.bigmodel.cn/api/paas/v4/`，其中包含了`zhipuai`的域名片段（`bigmodel.cn`是智谱的），代码会按照智谱AI的特定API格式（`model`, `query`, `documents`）来构造请求，并调用您指定的**`rerank`模型**。
