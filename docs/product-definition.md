# 智能文档问答与分析平台 · 产品定义

> 基于 RAG + AI Agent 的企业文档处理系统。核心解决一个问题：
> **"文档太多，找不到、看不完、分析不过来。"**

---

## 一、产品定位

### 一句话定义

上传文档，既能**问答找答案**，也能**自动分析出结论**。

### 两个核心能力

| 能力 | 一句话 | 解决谁的什么问题 |
|---|---|---|
| **A - 知识问答** | 问什么答什么，带来源 | 普通员工：文档太多找不到信息 |
| **B - 智能分析** | 说需求，自动读文档出结论 | 分析师：多份报告看不完、对比不过来 |

### 不是做什么

- ❌ 不是"企业 AI 中台"——不做模型训练、不做模型托管
- ❌ 不是"工作流编排平台"——工作流编辑是辅助能力，非核心卖点
- ❌ 不是"多模态"——只有文本能力，不处理图片/音频/视频

## 二、目标用户与场景

### 用户 A：普通员工

**痛点：** 公司知识库 / 内部 Wiki / 共享文件夹里的文档太多，搜不到想要的信息，或者搜到了也要翻半天。

**使用方式：**
1. 上传文档或接入已有文档库
2. 直接问："今年的年假政策是什么？"
3. 系统返回答案，附带引用来源（摘自哪份文档的哪个段落）

**典型问题：**
- "公司出差住宿报销标准是多少？"
- "Q3 的产品发布计划是什么？"
- "这个项目的技术架构用了哪些中间件？"
- "入职培训需要完成哪些课程？"

### 用户 B：分析师 / 研究员 / 法务

**痛点：** 需要读多份文档、对比差异、综合出结论，手动翻几十页 PDF 耗时且容易遗漏。

**使用方式：**
1. 上传多份文档（或指向已有文档）
2. 提出分析需求："对比这两份合同的风险条款差异"
3. Agent 自动规划步骤：分别读取两份文档 → 提取风险条款 → 逐条对比 → 输出结构化对比报告

**典型任务：**
- "对比 Q2 和 Q3 销售报告，找出业绩波动最大的区域"
- "总结这三份技术方案的优缺点对比"
- "读这五份简历，按技术能力排序并说明理由"
- "检查这份合同是否符合我们公司的标准条款模板"

### 用户 C：新员工

**痛点：** 入职需要阅读大量文档，无从下手。

**使用方式：**
- "我新入职后端组，需要看哪些文档？帮我总结每份的核心内容"

## 三、核心工作流

### 流程 A：知识问答（RAG 路径）

```
用户提问
   │
   ▼
┌─ 查询分析 ──────────────────────────────┐
│  QueryComplexityClassifier 判断复杂度     │
│  ├─ simple     → 仅关键词检索            │
│  ├─ medium     → 混合检索（默认）         │
│  └─ complex    → 混合 + HyDE + 查询改写  │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 双路检索 ───────────────────────────────┐
│  关键词检索（ES multi_match + BM25）      │
│  向量检索（ES dense_vector + cosine)      │
│  （复杂模式：HyDE 伪文档 + 多查询改写）    │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 结果融合 ───────────────────────────────┐
│  RRF（Reciprocal Rank Fusion, k=60）      │
│  MMR 多样性选择（λ=0.65）                 │
│  同文档去重（每文档最多 2 个块）           │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 重排序 ────────────────────────────────┐
│  Cross-Encoder Rerank（优先智谱 rerank） │
│  兜底：LLM 重排                          │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 生成回答 ──────────────────────────────┐
│  LLM + 检索上下文 → 带引用的回答          │
└──────────────────────────────────────────┘
   │
   ▼
  用户看到答案 + 来源引用
```

### 流程 B：文档分析（Agent 路径）

```
用户提出分析需求
   │
   ▼
┌─ Phase 0：记忆召回 ──────────────────────┐
│  AgentMemoryBridge 检索相关历史记忆       │
└──────────────────────────────────────────┘
   │
   ▼
┌─ Phase 1：规划（Planning）───────────────┐
│  Planner 将任务拆解为多步执行计划          │
│  例："对比两份合同"→                      │
│    步骤1：搜索合同A                       │
│    步骤2：搜索合同B                       │
│    步骤3：摘要合同A的风险条款             │
│    步骤4：摘要合同B的风险条款             │
│    步骤5：逐条对比输出结论                │
│  输出：结构化 Plan（依赖图）              │
└──────────────────────────────────────────┘
   │
   ▼
┌─ Phase 2：执行（Execution）──────────────┐
│  按依赖顺序执行每一步                      │
│  每一步：LLM 决策 → 调工具 → 观察结果     │
│  工具：知识库搜索 / 文档摘要 / 关键词提取  │
│        / 数据分析 / 翻译 / 联网搜索 等     │
│  失败自动重试（指数退避，最多 3 次）        │
│  中间结果存入 WorkingMemory               │
└──────────────────────────────────────────┘
   │
   ▼
┌─ Phase 3：反思（Reflection）─────────────┐
│  Reflector 评估执行结果质量               │
│  结果不满意 → 标记重试步骤                │
│  结果 ok    → 生成最终回答                │
└──────────────────────────────────────────┘
   │
   ▼
┌─ Phase 4：记忆存储 ──────────────────────┐
│  记录本次交互，积累经验供下次使用          │
└──────────────────────────────────────────┘
   │
   ▼
  用户看到分析结论 + 执行过程回溯
```

### 流程 A+B 混合：先检索后分析

```
用户提问 → RAG 检索相关文档
         → Agent 基于检索结果执行分析
         → 输出带文档引用的分析结论
```

## 四、RAG 管道详解

> RAG 是系统的"检索基础设施"，负责从海量文档块中找到最相关的内容。

### 4.1 文档入库

```
上传（PDF/Word/Excel/TXT）
   │
   ▼
┌─ 文档解析 ───────────────────────────────┐
│  PDF：pdfplumber（表格）+ pypdf（文本）    │
│       OCR 兜底（pytesseract）              │
│  DOCX：docx2txt                           │
│  Excel：openpyxl                          │
│  TXT：直接读取                            │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 语义分块 ───────────────────────────────┐
│  1. 正则切分句子                          │
│  2. 滑动窗口（buffer_size=1）组合上下文    │
│  3. Embedding 每个窗口                    │
│  4. 计算相邻窗口余弦距离                   │
│  5. 取第 90 百分位作为切分阈值            │
│  6. 丢弃 <100 字符碎片                    │
│  7. 合并 <1200 字符相邻块                 │
│  产出：语义完整的文本块                    │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 上下文增强（Contextual Retrieval）──────┐
│  1. LLM 生成文档摘要（50-100字）           │
│  2. 以「【文档背景：摘要】」前置到每块     │
│  目的：块独立检索时仍携带文档级上下文      │
└──────────────────────────────────────────┘
   │
   ▼
┌─ 向量化 + 索引 ──────────────────────────┐
│  Embedding 模型 → 2048 维向量             │
│  存入 ES dense_vector（HNSW 索引，cosine）│
│  同时存 MySQL（chunk 元数据）              │
└──────────────────────────────────────────┘
```

### 4.2 检索与生成

参见"流程 A：知识问答"。

### 4.3 质量保障机制

| 机制 | 作用 | 代码位置 |
|---|---|---|
| 语义分块 | 分块边界落在语义转折处，不切断完整意思 | `semantic_chunker.py` |
| Contextual Retrieval | 块携带文档级上下文，避免"断章取义" | `document_parser.py:_create_contextual_chunks()` |
| 查询复杂度自适应 | 简单问题不走向量检索，省钱省时间 | `query_processor.py:QueryComplexityClassifier` |
| HyDE + Multi-Query | 查询改写解决"问法和文档写法不一致" | `query_processor.py` |
| RRF 融合 | 关键词 + 向量双路结果排序融合 | `retriever.py:_rrf_fuse()` |
| MMR 多样性 | 避免返回结果同质化 | `retriever.py:_mmr_select()` |
| 同文档去重 | 每文档最多 2 个块，防止垄断结果 | `retriever.py:_post_process()` |
| Cross-Encoder Rerank | 对 Top-K 结果精确重排 | `reranker.py` |
| 语义缓存 | 相似查询直接命中缓存结果 | `cache.py` |
| 上下文压缩 | 控制输入 LLM 的 token 量 | `context_compressor.py` |
| RAG 评估指标 | 命中率 / 引用率 / 相关性评分 | `evaluator.py`, `metrics.py` |

## 五、Agent 详解

> Agent 是系统的"分析引擎"，负责执行多步推理和工具调用。

### 5.1 架构：PER（Plan-Execute-Reflect）循环

对比传统的 ReAct 模式：

| | ReAct | PER（本项目） |
|---|---|---|
| 规划 | 无，边想边做 | 先规划，再执行 |
| 执行 | 单步 | 多步，按依赖图执行 |
| 反思 | 无 | 执行完整体评估，可重试 |
| 记忆 | 无 | 持久化记忆 + 经验积累 |
| 子任务 | 无 | 支持子 Agent 委派 |

### 5.2 工具清单

| 工具 | 能力 | 场景 |
|---|---|---|
| `search_knowledge_base` | 关键词检索知识库 | 找文档 |
| `vector_search` | 语义搜索 | 找相关段落 |
| `summarize_document` | 全文摘要 | 读文档 |
| `extract_keywords` | 提取关键词 | 标签化 |
| `list_documents` | 列出文档 | 浏览 |
| `get_document_info` | 文档详情 | 查看元数据 |
| `list_conversations` | 历史对话 | 回溯上下文 |
| `get_conversation_history` | 对话详情 | 回溯上下文 |
| `list_prompt_templates` | 模板列表 | 查看模板 |
| `get_prompt_template` | 模板详情 | 查看模板 |
| `get_current_time` | 获取时间 | 时间相关查询 |

### 5.3 三级架构

```
Agent Service (service.py)
    │
    ▼
PER Agent Loop (loop.py)  ← 核心调度器
    │
    ├── Planner (planner.py)      → 任务拆解、生成 Plan
    ├── Executor (executor.py)    → 按依赖执行、工具调用、重试
    ├── Reflector (reflector.py)  → 反思评估、决定重试
    │
    ▼
Tool Registry (registry.py)   ← 插件式工具注册
    │
    ├── core_tools.py           → 基础工具（搜索、摘要等）
    └── tools/                  → 扩展工具
        ├── web_search.py       → 联网搜索
        ├── code_execution.py   → Python 代码执行
        ├── data_analysis.py    → 数据分析
        └── translation.py      → 翻译

Memory Bridge (memory_bridge.py)  ← 持久化记忆
Context Engine (context.py)       ← 上下文窗口管理
```

### 5.4 与 RAG 的关系

```
RAG  = 找得准（检索层）
Agent = 想得透（推理层）

两者关系：
  RAG 提供素材 → Agent 加工素材 → 输出结论

  也可以：
  Agent 在执行过程中，每一步需要信息时调用 RAG 工具搜索
```

## 六、A + B 结合的典型场景完整流程

### 场景："对比 Q2 和 Q3 销售数据"

```
用户输入："帮我对比 Q2 和 Q3 的销售报告，分析业绩波动"

  Step 1: Agent 规划
  └─ 规划器拆解：
     ① 搜索"Q2 销售报告"
     ② 搜索"Q3 销售报告"
     ③ 依次摘要两份文档
     ④ 提取关键数据指标
     ⑤ 对比分析 → 输出结论

  Step 2: Agent 执行（每步调 RAG 工具）
  ├─ search_knowledge_base("Q2 销售报告")
  │     → RAG 检索 → 返回文档 A
  ├─ search_knowledge_base("Q3 销售报告")
  │     → RAG 检索 → 返回文档 B
  ├─ summarize_document(A)
  │     → RAG 读取全文 → LLM 摘要 → 得到 Q2 数据
  ├─ summarize_document(B)
  │     → RAG 读取全文 → LLM 摘要 → 得到 Q3 数据
  └─ 综合对比 → 输出带引用的分析报告

  Step 3: Agent 反思
  └─ 检查结论完整性 → 如果有遗漏数据，标记重试

  Step 4: 输出
  └─ "与 Q2 相比，Q3 各区域业绩波动如下：
      华东 +15%（增长），华北 -8%（下降）...
      数据来源：Q2销售报告_第3页、Q3销售报告_第5页"
```

## 八、当前状态与边界

| 维度 | 状态 |
|---|---|
| **代码完整性** | 核心流程全部实现，仅 1 处 TODO（工作流的 TTS 节点） |
| **测试覆盖** | 273 个测试，25 个测试文件 |
| **Guest 体验** | guest/123456 账号自动初始化，Demo 数据可一键生成 |
| **并发能力** | Kafka 异步处理文档上传，支持多用户 |
| **多租户** | RBAC + Organization 隔离 |
| **部署** | Docker Compose 一键部署，K8s 清单就绪 |

### 已知不足

- **Agent 测试覆盖为 0** —— 273 个测试无一是 Agent 相关的
- **产品体验未闭环** —— Demo 数据需要手动调接口触发，不是首次访问自动触发
- **未做生产压测** —— 不知道高并发下的实际表现
- **无集成文档** —— 虽然有 Swagger，但没有面向集成方的 API 使用指南
